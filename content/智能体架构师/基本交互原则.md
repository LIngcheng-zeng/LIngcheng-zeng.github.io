初始静态协议：markdown文档

动态交互过程中：

1. 利用反向解释机制，确认任务逻辑闭环。
2. 颗粒度控制：不要试图一次性的完成需求到完整代码。要将大块的信息熵拆解为小的可控的单元，每一次都经过校验。

# 🤖 大模型精确交互增强版模板 (Advanced Prompt Template)

## 🎭 1. 角色设定 (Role)
- **身份**: 你是 [例如：高级资深程序员 / 资深雅思考官 / 商业文案专家]
- **专业领域**: 专注于 [例如：分布式架构 / 情感营销 / 法律合规]
- **交互语气**: [例如：专业严谨、平易近人、批判性思维、苏格拉底式启发]

## 📖 2. 任务背景 (Context)
- **目标场景**: [描述具体情境，如：我正在写一份给 CEO 看的季度汇报]
- **参考资料**: [在此粘贴已有信息、链接或数据摘要]
- **核心痛点**: [目前遇到的最大困难是什么]

## 🛠 3. 执行指令 (Task Instructions)
- **核心任务**: [清晰描述动作：总结、翻译、重构、扩写]
- **执行步骤**: 
  1. 第一步：[如：提取文中 5 个关键词]
  2. 第二步：[如：针对关键词进行利弊分析]
  3. 第三步：[如：生成最终结论]
- **思考路径**: 请在输出前执行“思维链（CoT）”，先列出你的分析框架。

## 🚫 4. 细化约束与边界 (Constraints & Boundaries)
> 请根据需要勾选或修改以下具体项：

### A. 语言与表达 (Language)
- [ ] **术语处理**: [例如：专业术语保留英文原词，不要翻译成中文]
- [ ] **禁忌词汇**: [例如：禁止使用“总之”、“综上所述”、“见证”、“赋能”等 AI 常用套话]
- [ ] **目标人群**: [例如：请用 5 岁小孩能听懂的语言解释 / 使用极具煽动性的营销口吻]

### B. 内容规范 (Content)
- [ ] **真实性控制**: [例如：如果你不确定事实，请直接回答不知道，严禁编造数据或案例]
- [ ] **信息屏蔽**: [例如：不要提及竞争对手的产品 A / 不要涉及任何政治敏感话题]
- [ ] **深度要求**: [例如：避免浅显的描述，请从底层原理/第一性原理出发进行深入分析]

### C. 技术与逻辑 (Technical)
- [ ] **代码/逻辑规范**: [例如：代码必须符合 Google 规范 / 优先考虑 O(1) 空间复杂度]
- [ ] **引用要求**: [例如：每一条结论都必须对应原文中的具体段落]
- [ ] **环境限制**: [例如：代码需在 Python 3.8 环境下运行，禁止使用第三方库]

### D. 格式与长度 (Formatting)
- [ ] **字数约束**: [例如：总字数控制在 200-300 字之间 / 摘要不超过 2 句话]
- [ ] **结构限制**: [例如：严禁使用 Markdown 二级标题 / 必须以 bullet points 形式呈现]
- [ ] **避开免责声明**: [例如：直接给出答案，不要在开头说“作为一个 AI 模型...”或在结尾加提示语]

## 📤 5. 输出格式要求 (Output Format)
- **首选格式**: [例如：Markdown 表格 / 标准 JSON 格式 / 代码块 / Mermaid 流程图]
- **输出模板**:
  ---
  ### [标题]
  - **核心逻辑**: [模型填充]
  - **具体建议**: [模型填充]
  ---

## 🧪 6. 示例 (Few-Shot)
- **输入**: [样例输入]
- **输出**: [理想的样例输出]

## ⚙️ 交互协议 — 三段式工作流

### 阶段一 · 反向解释
每个非极简任务从此处开始，输出任何方案前：
- 用自己的语言复述任务逻辑
- 列出 1–3 个模糊点，等待确认
- **硬性暂停** — 收到 `确认` 或 `NEXT` 后方可继续

### 阶段二 · 架构与接口设计
- 输出 Mermaid 流程图或 ASCII 架构草图
- 定义核心类签名、函数接口与数据模型
- 保持 Java 式严谨性：显式类型、清晰边界
- **硬性暂停** — 收到确认后方可继续

### 阶段三 · 分步实现与防御性编码
- 代码按逻辑模块分批输出，严禁整体倾倒
- 所有关键路径包含完整 try-catch 异常处理与结构化日志
- 代码统一英文

---
