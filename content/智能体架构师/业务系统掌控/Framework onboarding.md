# 🚀 开源框架快速上手 · 速查文档
> 场景：从零掌握一个陌生开源框架 | 预计投入：0.5 ~ 2h

---

## 🗂️ 阶段地图

```
[阶段 0] 定锚      →  [阶段 1] 扫描全局   →  [阶段 2] 核心概念
明确目标 & 边界        架构图 + 模块关系       3~5个必懂概念精讲

[阶段 3] 跑通示例  →  [阶段 4] 融入技术栈  →  [阶段 5] 边界探测
最小可运行Demo         与现有栈的集成路径       已知坑 & 版本陷阱
```

---

## 📋 每阶段 · Prompt 追问指令

### 阶段 0 · 定锚
```
目标：让模型理解你的学习边界，避免泛泛而谈。
追问示例：
  - "我的目标是：在 {{tech_stack}} 中使用 {{framework_name}} 完成 {{learning_goal}}，
     请只聚焦这个范围，不要展开其他场景。"
```

### 阶段 1 · 扫描全局
```
目标：获得一张架构鸟瞰图。
追问示例：
  - "用一张 ASCII 架构图描述 {{framework_name}} 的核心模块及其关系。"
  - "告诉我哪些模块是必须理解的，哪些是可选的。"
```

### 阶段 2 · 核心概念
```
目标：每次只攻一个概念，拒绝知识倾倒。
追问示例：
  - "用一个类比解释 {{concept}}，然后给我它在 {{framework_name}} 中对应的类/接口名。"
  - "这个概念和 Java 中的 {{java_equivalent}} 有什么本质区别？"
```

### 阶段 3 · 跑通示例
```
目标：最小可运行代码（非玩具，但也非生产级）。
追问示例：
  - "给我一个 {{learning_goal}} 的最小示例，技术栈是 {{tech_stack}}，
     版本是 {{version}}，不要引入额外依赖。"
  - "把上面的示例加上异常处理和日志，但保持代码量在50行以内。"
```

### 阶段 4 · 融入技术栈
```
目标：从 Demo 走向真实工程。
追问示例：
  - "在 {{tech_stack}} 项目中集成 {{framework_name}} 的标准姿势是什么？
     有无官方 Starter / AutoConfiguration？"
  - "配置项里哪3个是生产环境必须调整的？给出推荐值和原因。"
```

### 阶段 5 · 边界探测
```
目标：提前知道坑在哪。
追问示例：
  - "{{framework_name}} {{version}} 有哪些已知的 Breaking Change 或常见坑？"
  - "在高并发 / 大数据量场景下，{{framework_name}} 的哪个模块最容易成为瓶颈？"
```

## 📥 启动前物料清单

| 物料 | 是否必须 | 获取方式 |
|------|---------|---------|
| 框架名称 + 版本 | ✅ 必须 | 官网 / Maven Central |
| 当前技术栈描述 | ✅ 必须 | 自行填写 |
| 学习目标（一句话） | ✅ 必须 | 自行填写 |
| 时间预算 | 推荐 | 自行填写 |
| 官方文档 URL | 推荐 | 官网 |

---

## ⚠️ 常见失败模式

| 失败模式 | 症状 | 修正动作 |
|---------|------|---------|
| 目标模糊 | 模型输出大而全的概述 | 回到阶段0，重新定锚 |
| 跳过概念直接要代码 | 代码能跑但不理解 | 强制先走阶段2 |
| 一次追问太多 | 回答质量下降 | 每次只问一个问题 |
| 忽略版本 | 示例代码跑不通 | 每次 Prompt 都带上 `{{version}}` |

# 🏚️ 遗留业务系统解析 · 速查文档
> 场景：快速读懂一个你没写过的存量系统 | 预计投入：1h ~ 半天

---

## 🗂️ 阶段地图

```
[阶段 0] 物料收集    →  [阶段 1] 系统画像     →  [阶段 2] Git 考古
收集代码/DDL/日志       边界 + 模块 + 技术债       变更热点 + 演进脉络

[阶段 3] 核心流程    →  [阶段 4] 数据模型     →  [阶段 5] 风险评估
关键业务链路还原         表关系 + 字段语义          改动影响面 + 雷区标注
```

---

## 📋 每阶段 · Prompt 追问指令

### 阶段 0 · 物料收集（投喂前必做）

| 物料 | 获取命令 / 方式 | 是否必须 |
|------|--------------|---------|
| 核心类代码 | IDE 直接复制 | ✅ 必须 |
| 表结构 DDL | `SHOW CREATE TABLE xxx` | ✅ 必须 |
| Git 变更日志 | `git log --oneline -50` | ✅ 必须 |
| Git 高频变更文件 | `git log --name-only --pretty=format: \| sort \| uniq -c \| sort -rn \| head -20` | 推荐 |
| 口头业务描述 | 问业务方/PM | 推荐 |
| 当前痛点 | 自行描述 | 推荐 |

> ⚠️ Git Log 是核心物料，**绝对不要跳过**。变更频率是代码重要性的最真实信号。

---

### 阶段 1 · 系统画像
```
目标：获得一句话系统定位 + 边界。
追问示例：
  - "根据以下代码和描述，用一段话总结这个系统做什么、不做什么。"
  - "识别出系统的对外依赖（第三方服务/MQ/数据库），画出依赖关系图。"
  - "估计技术债的严重程度：1-5分，并说明依据。"
```

### 阶段 2 · Git 考古
```
目标：从提交历史读出系统演进脉络。
追问示例：
  - "分析以下 git log，找出变更最频繁的 3 个模块，推测它们是核心还是问题区域。"
  - "从提交信息中识别：有无重大重构记录？有无紧急修复（hotfix/urgent/fix）集中期？"
  - "高频变更文件列表如下，推测哪些是业务核心，哪些是长期补丁区。"

投喂格式：
  -----GIT_LOG_START-----
  {{git_log}}
  -----GIT_LOG_END-----
```

### 阶段 3 · 核心流程还原
```
目标：搞清楚最重要的一条业务链路。
追问示例：
  - "根据以下代码，还原 {{business_flow}} 的完整执行链路，用步骤编号列出。"
  - "这个流程里有哪些隐式的状态机？把状态和转换条件列出来。"
  - "哪个节点是最容易出故障的？原因是什么？"
```

### 阶段 4 · 数据模型解读
```
目标：读懂表结构背后的业务语义。
追问示例：
  - "根据以下 DDL，推断每张表的业务含义，重点解释命名模糊的字段。"
  - "表之间的关联关系是什么？有无缺少外键约束但实际存在逻辑关联的情况？"
  - "哪些字段的设计你认为存在风险（类型、长度、缺少索引等）？"

投喂格式：
  -----DDL_START-----
  {{ddl_schema}}
  -----DDL_END-----
```

### 阶段 5 · 风险评估
```
目标：改动前知道地雷在哪。
追问示例：
  - "如果我要修改 {{target_module}}，影响面可能涉及哪些模块？"
  - "从代码和表结构中，识别出 Top3 高风险改动点，说明原因。"
  - "{{pain_point}} 最可能的根因是什么？给出排查路径。"
```

## 🔍 Git Log 分析速查

```bash
# 获取最近 50 条提交（投喂给模型）
git log --oneline -50

# 找出变更最频繁的文件（识别核心/问题区域）
git log --name-only --pretty=format: | sort | uniq -c | sort -rn | head -20

# 查看某文件的完整变更历史
git log --follow -p -- path/to/file.java

# 找紧急修复集中期
git log --oneline --grep="fix\|hotfix\|urgent\|bugfix" --since="6 months ago"

# 按作者统计提交（识别核心负责人）
git shortlog -sn --since="1 year ago"
```

---

## ⚠️ 常见失败模式

| 失败模式 | 症状 | 修正动作 |
|---------|------|---------|
| 只给代码不给 DDL | 模型无法理解数据流 | 补充核心表的 DDL |
| 跳过 Git 考古 | 不知道哪里是真正的核心 | 先跑高频变更文件命令 |
| 一次粘贴太多代码 | 模型注意力发散，分析质量差 | 每次聚焦 1~2 个核心类 |
| 问题太宏观 | "帮我理解这个系统" | 缩小到具体流程或模块 |
| 忽略口头描述 | 代码和业务意图割裂 | 哪怕一句话背景也要写上 |

### 🔴 追问链 A · 状态机/流程深挖

**触发条件**：首轮输出含「状态流转 / 业务流程 / 链路 / 节点」等词

#### L1 · 还原状态全集
```
[SESSION_STATE]
...（填入状态头）
-----以上为上下文，本轮新问题如下-----

根据上面的分析，针对「{{business_flow}}」：
1. 列出该流程所有可能的状态值（包括中间态、异常态、终态）。
2. 用表格输出状态转换矩阵：| 当前状态 | 触发事件 | 目标状态 | 触发条件 |
3. 标出哪些状态转换缺少幂等保护。

结尾标注：【最高风险转换点】及原因。
```

#### L2 · 异常路径 & 补偿机制（依赖 L1 输出）
```

已确认结论:

  - 最高风险转换点: {{l1_risk_point}}
当前深挖节点: 链A / L2
-----以上为上下文，本轮新问题如下-----

针对【最高风险转换点】：
1. 这个节点失败后，系统当前的补偿逻辑是什么？（从代码中找证据）
2. 有无分布式事务保障？还是依赖本地事务？
3. 如果补偿失败，数据会停留在什么中间态？有无人工干预入口？

结尾标注：【补偿缺口等级】1~5分 + 依据。
```

#### L3 · 幂等性全面验证（依赖 L2 输出）

已确认结论:

-----以上为上下文，本轮新问题如下-----

全面评估该流程的幂等性：
1. 入口层（Controller/Consumer）有无重复请求过滤？

### 🔵 追问链 D · 数据模型深挖

## ⚠️ 失败模式速查

| 失败模式 | 症状 | 修正动作 |
|---------|------|---------|
| 忘记携带会话状态头 | 模型忘记上一轮结论，重复分析 | 每轮追问前复制状态头 |
| 一次投喂代码太多 | 分析质量下降，关键点被淹没 | 每次聚焦 1~2 个核心类 |
| 跳层追问 | L1 未完成直接跳 L3 | 严格按 L1→L2→L3 顺序 |
| 忽略【关键词标注】 | 不知道走哪条追问链 | 首轮 Prompt 强制要求模型输出关键词 |
| 监控数据未标注类型 | 模型无法正确解读指标含义 | 投喂时注明数据来源和类型 |
| 追问目标模糊 | 模型给泛泛建议 | 追问中锁定具体类名/表名/接口名 |
